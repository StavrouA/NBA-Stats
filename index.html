Abstract

The purpose of this document is to implement some basic Machine Learning classification algorithms, such as logistic regression, classification trees and support vector machines. We will check the accuracy of these algorithms without tuning them (for example, there was no test (e.g.Elbow method) ran for the selection of the number of k folds in k fold cross validation algorithm, I arbitrarily used k=10), and in the end, we will compare this accuracy to the one that we get from randomly guessing the outcome.

We notice that athletes who did not last in the NBA, have more of a leptokurtic distribution and are positively skewed (median is closer to 10 minutes per game), while players who lasted have a platykurtic distribution that is less skewed (median is closer to 20 minutes per game). From the 2nd graph, we get that people who lasted for more than 5 years in the NBA had a platykurtic distribution, the groups median had about 0.6 turnovers per match, while for people who didnt last the median has about 1 turnover per match.

Graphs 3 and 4 are two scatterplots with trend lines, showing the relationship of assists with turnovers and steals. Both turnovers and steals correlate with the average assist number per game. Higher turnover rate for athletes (esp.rookies) who attempt more assists on average was to be expected, due to taking higher risk when throwing a pass. However, the number of steals affecting the number of assists has no logical explanation, so maybe this correlation is spurious. However, we will not look into it, as this is not the purpose of this presentation (for those interested, follow this link: https://bit.ly/3av3m7X).

Splitting the dataset

Now, we split the dataset into training and testing sets:

set.seed(41)
indices = 1:nrow(nba.df)
trainIndex = sample(x = indices,size = floor(0.6*nrow(nba.df)),replace = F)
train = nba.df[trainIndex,]
temp_test = nba.df[-trainIndex,]

To be more unbiased, we break the temp_test dataset into a new testing and a cross validation test:

indices = 1:nrow(temp_test)
testIndex = sample(x = indices,size = floor(0.5*nrow(temp_test)),replace = F)
test = temp_test[testIndex,]
cv = temp_test[-testIndex,]

We now have 3 dataframes; test, train and cv, which consist of 60%, 20% and 20% of the original dataset respectively.

Predicting with LOOCV

Leave one out Cross Validation method works as follow: 1. Leave one data point out and train on the rest of the data 2. Test the model on the data point left out and record the prediction error 3. Repeat for all data points 4. Compute overall prediction error by averaging all predictions errors from step 2

require(caret)
set.seed(120)
# Define training control
train.control <- trainControl(method = "LOOCV")
# Train the model
loocv <- train(as.factor(TARGET_5Yrs) ~., data = train, method = "bayesglm",
               trControl = train.control)
# Summarize the results
print(loocv)
## Bayesian Generalized Linear Model

##   2 classes: '0', '1'

## Resampling results:
## 
##   Accuracy  Kappa    
##   0.699005  0.3455103
loocv_pred <- predict(loocv, newdata = cv)
table(predicted = loocv_pred, actual = cv$TARGET_5Yrs)
##          actual
## predicted   0   1
##         0  58  40
##         1  41 129
mean(loocv_pred == cv$TARGET_5Yrs)
## [1] 0.6977612

We now have lower accuracy than logistic regression. However, the positive about this algorithm is the fact that it tests the model on every data point, so the bias is expected to be lower. The con is that because of training and testing for every data point, it is way slower than using logistic regression.

Predicting with kfold cross-validation

To avoid the con mentioned above, we will predict by using kfold cross-validation. This algorithm works similarly to LOOCV, but instead of using every data points it uses k subsets of the dataset. We will repeat this process 3 times, so that the split of the subsets becomes more random.

set.seed(223)
train.control <- trainControl(method = "repeatedcv", 
                              number = 10, repeats = 3)
# Train the model
rkfold <- train(as.factor(TARGET_5Yrs) ~., data = train, method = "bayesglm",

Predicting with Random Forests

In this section we implement a tuned Random Forest algorithm. We set the seed and then train our model. We check which ntree number has the lower Out Of Bad (OOB) error rate, and then we train again our model with optimal ntree parameter:

require(randomForest)
set.seed(123)
rf <- randomForest(as.factor(TARGET_5Yrs) ~ ., data = train, ntree = 500, mtry = 6, importance = TRUE)
n <- which.min(rf$err.rate[,1])
set.seed(123)
rf <- randomForest(as.factor(TARGET_5Yrs) ~ ., data = train, ntree = n, mtry = 6, importance = TRUE)
rf_pred <- predict(rf, cv)

plot(rf, log="y", main = "Random Forest error rates")
legend("topright", colnames(rf$err.rate),col=1:4,cex=0.8,fill=1:4)

mean(rf_pred == cv$TARGET_5Yrs)
## [1] 0.7126866

We notice that our accuracy has is now 71.27%, which is lower than both SVM and logit, but it is higher than classification trees.

Final prediction

We split our dataset into 3 subsets: test (60%), cv(20%) and test(20%), so that the final predictions are as much unbiased as possible (we did not train any model so that it has higher accuracy in the testing set, we used cv instead). The highest accuracy was achieved by using logistic regression and support vector machines. So, the final predicitons should be made by using either one of these two trained models:

set.seed(555)
logit_prob_final <- predict(logit, newdata = test, type = "response")
logit_pred_final <- rep(0, dim(test)[1])
logit_pred_final[logit_prob_final > .5] <- 1

Conclusions

We saw that every algorithms improves our accuracy for about 8% to 9%. We got these results without fine-tuning our algorithms, meaning that we can probably get even higher percentages. Does this document imply that the best classification algorithms is logit or SVM? The answer is no. The fact that in this case we got better results off these two algorithms do not prove that, in fact, every case is different. The best algorithm could cary based on the number of features, observations, etc. Also, in many cases, it could be that we use simpler and computationally lighter algorithms over heavier ones, if the outcomes do not vary that much. As a reminder, always try simpler methods first, most of the times they produce satisfying results :)
